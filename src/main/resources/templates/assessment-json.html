JSON Input Guide

How structured JSON assessment works

JSON input is designed for users who want a strict, predefined, and reproducible assessment format. Instead of describing the organization in plain text, the user provides structured data about the organization, its threat context, implemented security controls, and defensive capabilities.

D3FENDer reads the JSON file, identifies the relevant sections, normalizes the provided values into internal defensive measures, retrieves the appropriate assessment rules, and evaluates whether important security capabilities are present or missing. If the conditions of a rule are met, the platform generates a security finding and maps it to relevant ATT&CK and D3FEND techniques.

Expected top-level structure

{
  "metadata": {
    "organization": "Mike Corp",
    "assessment_id": "assessment_01",
    "assessment_date": "15-03-2026",
    "assessor": "Michael"
  },
  "threat_context": {},
  "security_controls": {},
  "defensive_capabilities": {}
}

threat_context

Describes the organization’s environment, activities, and exposure. These fields help D3FENDer decide which rules are relevant before evaluating defenses.

  • software_engineering_activity
  • uses_web_applications
  • handles_sensitive_data
  • internet_facing_services
  • uses_cloud_services
  • remote_access_enabled

security_controls

Contains implemented security controls grouped by category. Values are usually boolean: true if implemented, false if missing.

  • network_security.firewall
  • network_security.vpn
  • access_control.identity_access_management
  • access_control.role_based_access_control
  • monitoring.logging
  • monitoring.alerting

defensive_capabilities

Represents detailed defensive capabilities. This is the most important section because it directly affects gap detection and rule evaluation.

  • authentication.token_based_authentication
  • authentication.password_authentication
  • authentication.multi_factor_authentication
  • authentication.certificate_based_authentication
  • credential_hardening.certificate_pinning
  • credential_hardening.credential_rotation

Example rule interpretation

For example, if the threat context shows that the organization performs software engineering activity, but defensive capabilities such as ASLR or stack frame canary validation are missing, D3FENDer can generate a finding such as missing stack overflow mitigations. The finding can then be linked to ATT&CK techniques like exploitation of public-facing applications and D3FEND techniques such as address randomization or stack canary validation.
